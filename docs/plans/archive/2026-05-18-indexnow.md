# Plan: IndexNow-støtte (Bing)

**Dato:** 2026-05-18
**Oppgave:** Varsle Bing (og andre IndexNow-søkemotorer) automatisk ved deploy

## Bakgrunn

IndexNow er en protokoll støttet av Bing, Yandex m.fl. Ved publisering sendes en HTTP POST til `api.indexnow.org` med en liste over endrede URL-er. Søkemotoren verifiserer eierskap via en nøkkelfil plassert på domenet (`/{key}.txt` med nøkkelen som innhold).

---

## Del 1 — GitHub Secret

Generer nøkkel lokalt:
```bash
openssl rand -hex 16
```
Nøkkelen er ren hex — trygt å bruke i filnavn og JSON-payload uten escaping.

Legg til som GitHub Secret med navn `INDEXNOW_KEY` under *Settings → Secrets and variables → Actions*.

---

## Del 2 — Verifikasjonsfil

### Tilnærming
Nøkkelfilen genereres **i `deploy`-jobben** rett etter at build-artefakten er lastet ned og **før `Generate CSP hashes from build output`-steget**. Da trenger ikke nøkkelen inn i build-jobben, og filen landes i S3 som en del av det ordinære "Alt annet"-synksteget.

```yaml
- name: Create IndexNow key file
  env:
    INDEXNOW_KEY: ${{ secrets.INDEXNOW_KEY }}
  run: |
    if [ -z "$INDEXNOW_KEY" ]; then
      echo "INDEXNOW_KEY not set — skipping"
      exit 0
    fi
    echo -n "$INDEXNOW_KEY" > dist/${INDEXNOW_KEY}.txt
```

`echo -n` sikrer ingen trailing newline, som IndexNow-spec krever. Filen deployes til S3 og er tilgjengelig på `https://www.tennerogtrivsel.no/{key}.txt`.

### Nøkkelrotasjon
Hvis `INDEXNOW_KEY` byttes ut, vil gammel `{old-key}.txt` bli liggende i S3 (S3-synken sletter bare filer som ikke finnes i `dist/`). Gammel fil må slettes manuelt fra S3-bucketen etter rotasjon.

## Del 3 — IndexNow-kall i `deploy`-jobben

### Plassering
Steget plasseres **etter `Invalidate CloudFront cache PROD`**-steget — etter at innholdet er deployet til S3. Merk: `aws cloudfront create-invalidation` er asynkront og returnerer umiddelbart; selve cache-tømmingen propagerer over alle edge-noder i løpet av 1–10 minutter. IndexNow-kallet sendes altså mens propagering pågår, men det er OK — søkemotorens crawler ankommer ikke umiddelbart uansett.

### URL-liste
Sitemap-artefakten (`dist/sitemap-0.xml`) er allerede tilgjengelig i `deploy`-jobben (lastet ned via `actions/download-artifact`). Ingen nettverkskall nødvendig.

`@astrojs/sitemap` sharding-grense er 45 000 URL-er per shard. Med nåværende URL-antall vil alt ligge i `sitemap-0.xml`. Hvis antallet nærmer seg 45 000 må steget oppdateres til å iterere alle shards fra `sitemap-index.xml`.

### Steg

```yaml
- name: Notify IndexNow
  continue-on-error: true
  env:
    INDEXNOW_KEY: ${{ secrets.INDEXNOW_KEY }}
  run: |
    if [ -z "$INDEXNOW_KEY" ]; then
      echo "INDEXNOW_KEY not set — skipping"
      exit 0
    fi

    URLS=$(grep -oP '(?<=<loc>)[^<]+' dist/sitemap-0.xml | jq -R . | jq -s .) || true

    if [ -z "$URLS" ] || [ "$URLS" = "[]" ]; then
      echo "No URLs found in sitemap — skipping IndexNow"
      exit 0
    fi

    PAYLOAD=$(jq -n \
      --arg host "www.tennerogtrivsel.no" \
      --arg key "$INDEXNOW_KEY" \
      --arg keyLocation "https://www.tennerogtrivsel.no/${INDEXNOW_KEY}.txt" \
      --argjson urlList "$URLS" \
      '{host: $host, key: $key, keyLocation: $keyLocation, urlList: $urlList}')

    curl -Sf -X POST "https://api.indexnow.org/indexnow" \
      -H "Content-Type: application/json" \
      -d "$PAYLOAD"
```

**`|| true`** etter grep-pipeline forhindrer at tomt resultat avbryter steget (GHA kjører bash med `set -eo pipefail`).

**`jq -n` for payload** eliminerer shell-interpolasjonsfeil og produserer alltid gyldig JSON.

**`-Sf`** (stor S) på curl viser feilmeldinger i loggen ved HTTP-feil, uten progress bar ved suksess.

**`continue-on-error: true`** — et feilende IndexNow-kall er ikke kritisk og skal ikke blokkere deploys. Feil logges i GitHub Actions.

## Del 4 — Bing Webmaster Tools (anbefalt, gjøres ved implementering)

Registrer `www.tennerogtrivsel.no` i [Bing Webmaster Tools](https://www.bing.com/webmasters). Dette er den eneste måten å bekrefte at IndexNow-innsendinger mottas og behandles. IndexNow fungerer teknisk uten dette, men uten registrering er det ingen synlighet i om det faktisk virker.

## Cache-control for nøkkelfilen

Nøkkelfilen arver "Alt annet"-policyen (`max-age=3600, stale-while-revalidate=86400`). Det er tilstrekkelig — filen endres bare ved manuell nøkkelrotasjon, som uansett krever et manuelt S3-opprydningssteg.
